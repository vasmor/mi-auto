# Правила интеграции в WordPress

**Назначение:** свод правил для тем WordPress/WooCommerce: структура кода, метаполя, шаблоны, подключение скриптов и стилей. Применимы к любой теме WordPress.

---

## 1. WordPress Coding Standards

### 1.1. Отступы и форматирование

- Один tab (4 пробела) для отступов
- Пробелы внутри скобок и вокруг операторов
- Один оператор на строку
- Максимально читаемый код

### 1.2. Именование

- **Файлы:** kebab-case (`prices-page.php`, `theme-btn.css`)
- **Функции и переменные:** snake_case с уникальным префиксом темы (например, `theme_slug_get_data()`, `$theme_slug_url`)
- **Классы:** StudlyCaps с префиксом (например, `Theme_Slug_Component`)
- **Константы:** UPPER_CASE (`THEME_SLUG_VERSION`)

### 1.3. PHPDoc

Добавлять PHPDoc к функциям и классам:

```php
/**
 * Краткое описание функции.
 *
 * @param string $param Описание параметра.
 * @return string Описание возвращаемого значения.
 */
```

### 1.4. Теги PHP

Использовать только полные теги `<?php ?>`. Не использовать короткие `<? ?>`.

## 2. Подключение стилей

### 2.1. Общие принципы

- Использовать `wp_enqueue_style()` для добавления CSS в очередь
- Регистрировать стили через `wp_register_style()` при необходимости переиспользования
- Указывать зависимости (например, если стиль зависит от другого)
- Использовать `get_template_directory_uri()` или `get_stylesheet_directory_uri()` для путей

### 2.2. Базовые стили

- Подключать в `wp_enqueue_scripts` (приоритет по умолчанию)
- Базовый reset, типографика, переменные — загружать на всех страницах
- Указывать `array()` как зависимости, если базовые стили не зависят от других

### 2.3. Стили секций

- Каждой секции — свой handle и файл
- Подключать стиль секции **условно**: только когда секция выводится на странице
- Вызов `wp_enqueue_style()` — в шаблоне секции или в логике, которая решает о выводе секции
- Если секция подключается через `get_template_part()`, стили можно подключать в начале файла секции с проверкой `$args['skip_styles']` для гибкости

### 2.4. Критические стили

- Секции в пределах первых ~1200px — подключать в critical CSS (в `<head>` до основного контента)
- Реализация: либо inline critical CSS, либо отдельная очередь с `wp_add_inline_style` или своя логика вывода в header

### 2.5. Порядок подключения

1. Базовые стили (reset, переменные, типографика)
2. Общие компоненты (кнопки, формы)
3. Стили секций (в порядке вывода на странице)

### 2.6. Версионирование

Передавать версию в `wp_enqueue_style()` для инвалидации кэша при обновлении (константа темы, `filemtime()` и т.д.).

## 3. Подключение скриптов

### 3.1. Общие принципы

- Использовать `wp_enqueue_script()` для добавления JS в очередь
- Регистрировать скрипты через `wp_register_script()` при переиспользовании
- Указывать зависимости (jQuery, библиотеки)
- Для скриптов в footer — последний аргумент `true`

### 3.2. Базовый скрипт темы

- Подключать в `wp_enqueue_scripts`
- Общие обработчики: меню, модальные окна, lazy loading
- Зависимости: при необходимости jQuery (`array( 'jquery' )`)

### 3.3. Скрипты секций

- Каждой интерактивной секции — свой файл и handle
- Подключать **условно**: только когда секция выводится
- Вызов `wp_enqueue_script()` — в шаблоне секции или в логике вывода
- Зависимости: указать общие библиотеки (Swiper, Fancybox и т.д.), если скрипт секции их использует

### 3.4. Ленивая загрузка тяжёлых библиотек

- Для секций ниже первого экрана: не подключать библиотеку сразу
- Использовать единый менеджер ресурсов: при появлении секции в viewport или при первом взаимодействии пользователя — загрузить библиотеку и выполнить инициализацию
- Регистрация скрипта через `wp_register_script()`; `wp_enqueue_script()` — по условию или через свой механизм (добавление `<script>` в DOM при необходимости)

### 3.5. Передача данных в скрипты

- Использовать `wp_localize_script()` для передачи в JS URL темы, строк, настроек
- Избегать inline-скриптов в шаблонах; выносить конфигурацию в `wp_localize_script` или data-атрибуты

### 3.6. Порядок подключения

1. Библиотеки (если загружаются сразу)
2. Базовый скрипт темы
3. Скрипты секций (по мере вывода секций)

## 4. Метаполя (Carbon Fields)

### 4.1. Типы полей

- **Заголовок:** `text`. При подсветке части — хранить полный текст, подсвечиваемая часть — отдельным полем
- **Текст/Описание:** `rich_text` для форматированного текста
- **Изображение:** `image`
- **Кнопка:** группа полей (текст, ссылка, возможно стиль) или переиспользуемая функция
- **Список элементов:** `complex`
- **Связи:** `association` для постов и терминов
- **Настройки:** `checkbox`, `select`, `text` (number) для опций

### 4.2. Условная логика

Использовать условное отображение полей: показывать поле только при определённом значении другого поля (например, «фильтр = по услугам» → показать выбор услуг).

### 4.3. Подсказки

Добавлять help text к полям, где нужно пояснение («Пусто — используется значение по умолчанию» и т.п.).

## 5. Шаблоны секций

### 5.1. Получение данных

- Опции темы: функция получения theme options (в Carbon Fields — `carbon_get_theme_option()`)
- Post meta: по ID записи
- Term meta: по ID термина для таксономий

### 5.2. Экранирование и вывод

- Обычный текст: `esc_html()`, `esc_attr()`
- Rich text: функция форматирования или `wp_kses_post( wpautop( $text ) )`
- SVG: `wp_kses()` с разрешёнными тегами
- Изображения: `wp_get_attachment_image()` с `loading="lazy"`
- URL: `esc_url()`

### 5.3. Заголовки с подсвечиваемой частью

- Хранить: полный текст + подстрока для подсветки
- Выводить через функцию, которая ищет подстроку в полном тексте и оборачивает её в `<span class="...">`
- Класс акцента — элемент блока: `section-name__title-accent`

### 5.4. Ранний выход

При отсутствии обязательных данных — `return` в начале шаблона, чтобы секция не выводилась.

### 5.5. Аргументы шаблона

При подключении через `get_template_part()` передавать массив аргументов (например, `post_id`, `term_id`, `skip_styles`) для гибкой настройки вывода и подключения ресурсов.

## 6. Структура классов в разметке

- Корень секции: блок (например, `section-name`)
- Контейнер: единый класс `.container` (определён в `css/base.css`), дополнительные стили задаются через `.section-name .container {}`
- Исключение: если контейнер — это оформленная карточка с `margin-left/right` вместо padding (например, `form-section__container`, `top-bar__strip`) — сохранять BEM-имя
- Элементы: `section-name__title`, `section-name__content` и т.д.
- Модификаторы: отдельный класс с ведущим тире (` -open`, ` -variant`)

---

## 7. Регистрация и очередь ресурсов

### 7.1. Централизованная регистрация

- Все стили и скрипты регистрировать в одном месте (отдельный файл `inc/enqueue/` или `functions.php`)
- Регистрация — через `wp_register_style()` и `wp_register_script()` при инициализации темы
- Подключение — через `wp_enqueue_style()` и `wp_enqueue_script()` в нужный момент (глобально или условно)

### 7.2. Роль регистрации

- Регистрация создаёт «запись» о ресурсе с handle, URL, зависимостями, версией
- Подключение добавляет ресурс в вывод текущей страницы
- Один раз зарегистрированный ресурс можно подключать в разных местах по handle

### 7.3. Зависимости

- Указывать корректные зависимости, чтобы WordPress выводил ресурсы в правильном порядке
- При отсутствии зависимости скрипт/стиль может выполниться раньше необходимого
